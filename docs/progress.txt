# Polet AI — Progress

## Status Summary

Polet AI is being redirected from a public policy smart wallet into a single-contract confidential multichain agentic wallet for AI agents.

The old v1 work remains useful as prior art, but the final hackathon path may substantially refactor the contract, proxy, SDK, and frontend. The key privacy requirement is that plaintext numeric policy rules must not remain in the final confidential execution path.

Primary source of truth:

- PRD: `docs/prd.md`
- Local issues: `docs/issues/*.md`
- Agent prompt: `docs/PROMPT.md`

## Product Direction

**Polet AI Confidential Multichain Agentic Wallet**

Users deposit funds into a Polet smart wallet PDA, configure confidential numeric policy rules, grant an AI agent a temporary session key, and let the agent run guarded strategy intents. The current Solana path uses Jupiter for token intelligence, pricing, and route/build previews. The target ending is a multichain agentic wallet where the same private guardrails can approve bridgeless native-asset trading requests through Ika.

Core demo:

- DCA run: 5 USDC -> SOL
- Confidential max per run: 10 USDC
- Confidential daily cap: 20 USDC
- Block scenario: 25 USDC
- Allow scenario: 5 USDC

Track alignment:

- Encrypt: confidential numeric policy enforcement is core.
- Ika: target execution rail for multichain bridgeless trading requests after guardrail approval.
- Jupiter: Tokens + Price + Recurring compatibility + Swap V2 `/build` fallback.
- National Campus Hackathon Indonesia: Consumer App / DeFi safety for Indonesian users.

## Done

### Current Roadmap

- [x] `001-confidential-smart-wallet-core` — Refactored the single contract core into a PDA wallet with owner authority, proxy metadata, policy commitment, Merkle root metadata, policy sequence, kill-switch revocation slot, and session-key records. Removed plaintext `daily_limit`, `daily_spent`, and serialized plaintext policy fields from the contract state and final session execution path. Synced the proxy IDL to the new contract ABI. Added tests for initialization, policy/proxy metadata, grant, revoke, revoke-all, expired/missing/revoked sessions, valid session execution, and stale policy sequence.
  - Verification: `NO_DNA=1 cargo build`, `NO_DNA=1 anchor build`, and `NO_DNA=1 cargo test` all pass in `contract/`.
- [x] `002-confidential-numeric-policy-enforcement` — Added the first pre-alpha confidential numeric policy path to the single contract. Wallet state now stores masked numeric policy values plus a commitment and witness hash instead of plaintext max-per-run, daily-cap, or daily-spent fields. Added owner policy setup and session execution gated by max-per-run, daily-cap, daily-spent update, and UTC-day reset behavior. Synced the proxy IDL to the new contract ABI.
  - Verification: `NO_DNA=1 cargo build`, `NO_DNA=1 anchor build`, and `NO_DNA=1 cargo test` all pass in `contract/`; `bun test` passes in `proxy/`.
  - Note: this is an Encrypt pre-alpha interface simulation using masked values and a witness hash. It does not claim production-grade confidentiality until real Encrypt primitives replace the witness path.
- [x] `003-pda-token-custody-usdc-sol-dca` — Added PDA-owned demo token custody metadata to the single wallet state plus an owner-only `register_demo_custody` instruction. The contract records USDC mint/account and SOL/wSOL mint/account references only after validating that the supplied SPL Token accounts are owned by the wallet PDA authority, so deposit addresses can be surfaced by proxy/frontend without putting funds under direct owner authority. Synced the proxy IDL to the new ABI.
  - Verification: `NO_DNA=1 anchor build` and `NO_DNA=1 cargo test` pass in `contract/`; `bun test` passes in `proxy/`.
  - Note: this slice establishes custody references and validation. Token movement from the PDA-owned accounts remains intentionally deferred to the Jupiter/confidential execution slices.
- [x] `004-jupiter-strategy-gateway` — Added an isolated proxy Jupiter strategy gateway for Tokens V2 metadata/quality fields, Price V3 USDC/SOL pricing, Recurring compatibility recording, and Swap API V2 `/build` fallback with the Polet wallet as taker. The gateway validates `JUPITER_API_KEY` before requests, supports injected fetch for deterministic tests, and captures Jupiter DX notes for the final report.
  - Verification: `bun test` and `bun run build` pass in `proxy/`.
  - Note: Recurring is recorded as incompatible for the current MVP path because Polet still needs every DCA run to pass confidential PDA policy gating immediately before spending; Swap V2 `/build` remains the execution fallback.
- [x] `005-confidential-dca-execution-path` — Added the proxy confidential DCA run path at `POST /intent/dca/run`, connecting wallet/session validation, Jupiter Tokens/Price/Swap V2 prechecks, pre-alpha confidential max-per-run and daily-cap witness evaluation, non-leaking block responses, and unsigned confidential session transaction construction. The allowed path uses the smart wallet PDA as Jupiter taker/payer and Polet contract wallet authority; the proxy does not sign or broadcast.
  - Verification: `bun test` and `bun run build` pass in `proxy/`.
  - Note: this remains an Encrypt pre-alpha masked witness demo path, not a production confidentiality guarantee. Jupiter Recurring remains incompatible for the MVP, so Swap V2 `/build` fallback is used.
- [x] `006-agent-sdk-strategy-intents` — Extended the SDK with `createDcaIntent`, `createRiskGatedSwapIntent`, proxy `submitIntent` and `evaluateIntentWithProxy` helpers, Jupiter USDC/SOL constants, typed proxy errors, and OpenClaw/Hermes-style agent examples. Existing transfer, swap, stake, unstake, delegate, undelegate, and custom builders remain available.
  - Verification: `bun test` and `bun run build` pass in `sdk/`.
  - Note: DCA submission targets the issue 005 proxy route `POST /intent/dca/run`; risk-gated swaps intentionally retain action `swap` for backward compatibility with the existing proxy evaluator.

## Notes

- The final product uses one contract. Do not create a separate `polet_v2` program.
- Encrypt pre-alpha must be described honestly. Do not claim production confidentiality.
- Jupiter `/build` is preferred for smart wallet execution because Polet needs raw instruction control.
- Jupiter Recurring is primary only if compatible with the smart wallet flow; otherwise use Polet scheduler + Swap V2 `/build`.
- Ika is the target bridgeless execution rail for multichain native-asset trading requests. Do not claim real Ika settlement until an actual Ika backend/request path is verified.
- Encrypted allowlist/blocklist membership is out of scope for the first milestone.
